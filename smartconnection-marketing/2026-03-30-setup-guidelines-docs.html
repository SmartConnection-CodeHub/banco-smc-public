🚀 Setup Inicial de Nuevo Proyecto
Proceso paso a paso. Cada agente ejecuta su fase en orden. Tiempo estimado: 1-2 sesiones.
1
Kickoff & Priorización
👔 PM 💼 Comercial
PM define la North Star metric del proyecto. Comercial valida la demanda de mercado y propuesta de valor. Se crea la carpeta del proyecto en ~/Claude/Proyectos/[nombre]/ con subcarpetas por agente.
📄 PM: roadmap-inicial.xlsx · Comercial: propuesta-valor.docx
2
Discovery & Benchmark
🐕 Panchita 💼 Comercial ⚡ Sergito
Panchita investiga 3-5 competidores, mapea procesos BPMN, define user stories. Comercial aporta battle cards del mercado. Sergito cuestiona supuestos y propone enfoques no obvios.
📄 Panchita: benchmark.html · bpmn-procesos.html · user-stories.md · Sergito: ideas-divergentes.md
3
Diseño Funcional & Maqueta
🐕 Panchita
Maqueta HTML interactiva con datos reales, tokens CSS, todos los estados (loading, error, vacío, éxito). Modelo de datos, contratos API, reglas de negocio, SLAs. Versión desktop Y mobile.
📄 Panchita: maqueta-[nombre].html · modelo-datos.md · contratos-api.md · checklist-handoff.md
4
Arquitectura Técnica
🔍 Arielito 🔌 Integrador
Arielito crea ADR, define API contracts entre ABAP y Fiori, estructura de repos, estándares. Integrador identifica qué APIs externas se necesitan, selecciona providers IA, define conectores. Se crean los repos en GitHub (frontend + backend separados).
📄 Arielito: adr-001.md · api-contracts.md · tech-design.md · Integrador: catalogo-apis.xlsx · providers-ia.md
5
Implementación (paralelo)
🐾 Hoku 🔧 ABAP 🎨 Fiori 🛠️ Pipeline
ABAP implementa backend (APIs, DB, auth). Fiori implementa frontend (UI, responsive, SEO). Hoku implementa features de IA (prompts, RAG, agentes). Todos contra el API contract de Arielito. Pipeline configura CI/CD, env vars, auto-deploy. Trabajan EN PARALELO — el contrato es el pegamento.
📄 ABAP: api-docs.md · Fiori: components.md · Hoku: ai-specs.md · Pipeline: ci-cd-config.yaml
6
QA & Verificación
👩 Camilita 🔍 Arielito
Camilita testea como usuario real (6 personas, todos los breakpoints, pruebas destructivas). Arielito verifica cumplimiento de arquitectura, scoring A-F, deuda técnica. Bugs → ABAP/Fiori/Hoku corrigen. UX confusa → Panchita rediseña.
📄 Camilita: test-report.xlsx · bugs.md · Arielito: audit-report.md · debt-tracker.xlsx
7
Launch & Post-Launch
🛠️ Pipeline 🎧 ClienteX 💼 Comercial 👔 PM
Pipeline deploya a producción con smoke test. ClienteX prepara onboarding, knowledge base, SLAs de soporte. Comercial activa canales de venta. PM mide North Star metric y planifica siguiente sprint.
📄 Pipeline: deploy-log.md · ClienteX: onboarding-guide.docx · PM: sprint-review.xlsx
⚡ Cómo Trabajar Eficientemente con 12 Agentes
Reglas para no perder tokens ni tiempo.
🎯
1 tarea = 1 agente
No invocar múltiples agentes para lo mismo. Identificar el agente correcto y asignar directo. Si no sabes cuál → PM decide.
⚡
Paralelizar siempre
ABAP + Fiori + Hoku trabajan en paralelo contra el API contract. No esperar que uno termine para empezar otro.
📐
Contrato antes que código
Arielito define API contracts ANTES de implementar. Sin contrato = sin paralelismo = desperdicio.
📊
Scope por tarea
Fix visual → Fiori solo (1%). Feature → Panchita+Arielito+ABAP+Fiori (5-10%). Producto nuevo → todos (15-20%).
🔄
Feedback inmediato
Camilita testea APENAS se deploya, no al final del sprint. Bugs frescos = correcciones baratas.
📁
Documentar en carpeta
Cada agente guarda su output en ~/Claude/Proyectos/[proyecto]/[Agente]/. No en Slack, no en aire.

📐 Guidelines de Desarrollo — Agentes TI
Estándares que ABAP, Fiori, Hoku, Integrador y Pipeline siguen. Definidos por Arielito.
🔧 TypeScript
  • Strict mode en todos los proyectos
  • No any — tipos explícitos siempre
  • Interfaces para API contracts
  • Enums para valores finitos
  • Exhaustive checks en switches
📝 Código
  • Funciones <30 líneas
  • Archivos <300 líneas
  • camelCase vars, PascalCase components
  • kebab-case archivos
  • Imports: externos → internos → tipos
  • No comentarios obvios
🔌 APIs
  • RESTful con verbos HTTP correctos
  • Versionado: /api/v1/...
  • Paginación cursor-based
  • Error format: { error: { code, message } }
  • Rate limiting por endpoint
  • OpenAPI spec documentada
🗄️ Base de Datos
  • Migrations versionadas (nunca db push prod)
  • RLS siempre activo en Supabase
  • snake_case tablas y columnas
  • Índices documentados y justificados
  • Datos seed realistas para dev
🔒 Seguridad
  • Secrets en env vars, NUNCA en código
  • Headers: CSP, HSTS, X-Frame-Options
  • CORS solo orígenes permitidos
  • Input sanitization en boundaries
  • No PII en logs
  • Supabase service key SOLO para admin scripts
🎨 Frontend
  • Mobile-first siempre
  • Breakpoints: 375, 768, 1024, 1440
  • Touch targets mínimo 44x44px
  • Tokens CSS de Panchita (no hardcodear)
  • WCAG AA accesibilidad
  • Lighthouse >90
🐾 IA
  • Prompts versionados en código
  • Guardrails pre Y post en producción
  • Fallback chain mínimo 2 providers
  • Max tokens por request definido
  • Rate limit por usuario
  • NUNCA PII al modelo
  • Evals antes de deploy
  • LLMOps logging en cada llamada
🛠️ Deploy & CI/CD
  • push a main = producción (auto-deploy)
  • Build debe pasar antes de merge
  • Smoke test post-deploy obligatorio
  • Rollback en <15 min
  • Repos frontend y backend separados
  • S3 con versionado siempre
📋 Git
  • Branch: main (prod), develop (staging), feature/*
  • Commits descriptivos en español
  • No push --force a main
  • No amend sin pedir
  • Git pull antes de cerrar sesión
  • PR para features, direct push para hotfix
📐 Guidelines Funcionales — Agentes No-TI
Estándares que PM, Comercial, Panchita, Camilita, Sergito y ClienteX siguen.
👔 PM — Gestión
  • Todo item tiene RICE score antes de entrar al sprint
  • Máximo 3 items por sprint
  • North Star definida por proyecto
  • Reporte semanal con métricas AARRR
  • Roadmap trimestral actualizado mensualmente
💼 Comercial — Ventas
  • Discovery SPIN antes de propuesta
  • BANT para calificar, MEDDIC para deals grandes
  • Propuesta en <48h post-discovery
  • Win/Loss analysis en cada deal cerrado
  • Pipeline actualizado semanalmente
  • Battle cards por competidor
🐕 Panchita — Producto
  • Benchmark ANTES de diseñar (3-5 competidores)
  • BPMN para procesos cross-sistema
  • Maqueta HTML autocontenida con datos reales
  • Versión desktop Y mobile
  • Copy real, NUNCA placeholders
  • Checklist de handoff completa
👩 Camilita — QA
  • Testear como persona concreta (6 personas)
  • Todos los breakpoints: 375, 768, 1024, 1440
  • Bug report completo: severidad, pasos, esperado vs actual
  • Validar contra specs de Panchita
  • Regresión acumulativa (nunca el mismo bug 2 veces)
⚡ Sergito — Innovación
  • Mínimo 3 ideas NO obvias
  • Siempre cuestionar el brief primero
  • Conectar con dominio externo (biología, música, etc.)
  • Siempre primer paso accionable
  • Diverger antes de converger
🎧 ClienteX — Soporte
  • SLA: Crítica <1h, Alta <4h, Media <8h
  • Todo feedback categorizado y canalizado
  • Señales de churn → contacto proactivo
  • NPS survey trimestral
  • Knowledge base actualizada cada release

Convenciones de Nombres
Archivos: kebab-case siempre. Ejemplo: propuesta-infopet-2026-03.docx
Con fecha: YYYY-MM-DD al inicio. Ejemplo: 2026-03-30-benchmark-competidores.html
Con versión: v[N] al final. Ejemplo: prompt-chat-v3.json
Sprints: sprint-[N]. Ejemplo: sprint-14-planning.xlsx
Maquetas: También en ~/Downloads/maquetas/[proyecto]/ (doble-click para abrir)
