⚠️ Punto crítico de escala

Airtable Free tiene 1.000 registros máx por base. Con 228 registros fijos + 36 instancias diarias generadas por cron, la base se llena en ~21 días de uso real. Migración obligatoria a Airtable Team antes de operar en serio (~$20-100 USD/mes según número de seats). Alternativa: migrar capa de datos a Supabase Postgres (500MB free, 50K MAUs) cuando el costo lo justifique.

04 · Dimensiones

Hallazgos por Agente

Crítico

Colapso DB mes 7–10

Respuestas_Checklist: 36 plantillas × 19 usuarios × 365 días × ~10 ítems = 240,000+ registros/año. Límite Airtable Team: 125,000 filas/base. Colapso mes 7–10 sin migración planificada.

Crítico

8 ADRs faltantes

Decisiones arquitectónicas críticas sin documentar: Airtable vs DB relacional, NextAuth beta en prod, Vercel Hobby como hosting de negocio, cron via API route, Airtable Free en producción.

Alto

Lógica financiera sin versionado

Las fórmulas de Airtable (ventas − comisiones − mermas − subarriendo) son lógica de negocio crítica sin control de versiones, sin historial, sin rollback. Un cambio accidental destruye los cálculos de pagos.

Alto

Observabilidad 2/10

No hay Sentry, no hay logging estructurado, no hay métricas de negocio. El sistema es ciego en producción. El primer aviso de falla llega por WhatsApp de un usuario.

Alto

Sin idempotencia en cron

El cron de 7AM genera checklists sin verificar si ya existen. Un retry de Vercel crea duplicados que los usuarios ven como tareas nuevas. Sin mecanismo de deduplicación documentado.

Positivo

Stack apropiado para la escala

Next.js + Airtable + Vercel es la decisión correcta para 15–19 usuarios con el presupuesto disponible. La propuesta de migrar a Supabase + AWS implicaría 3–4 semanas de reescritura sin ganancia funcional real.

IDOR Confirmado — Insecure Direct Object Reference

Los 10 API routes validan autenticación (usuario logueado) pero no validan autorización (qué datos puede ver). Un Trabajador autenticado llama a GET /api/checklists?departamento=finanzas y obtiene datos de Finanzas. El roadmap de Créalo lo confirma como "P1 pendiente" — sabían de la vulnerabilidad y eligieron deployar con ella.

Crítico

Cron timeout 2.25× sobre límite

36 plantillas × 220ms = 7.92s en escenario base. Worst case: 16.2s vs límite Vercel Hobby de 10s. El cron falla silenciosamente sin alerta, sin retry, sin notificación al supervisor.

Crítico

Airtable API v0 Deprecated

La documentación menciona Airtable API v0 como dependencia. Esta versión está deprecated — Airtable puede eliminar el soporte sin aviso. SDK 0.12.2 es estable pero si hay código usando la API REST directamente, es deuda activa.

Alto

NextAuth v5 beta.31 en producción

NextAuth v5 estaba en beta al momento del deploy. Un breaking change en beta.32+ puede invalidar las sesiones de todos los usuarios simultáneamente. La versión estable v5 existe desde diciembre 2024.

Alto

Sin transacciones compensatorias

El flujo Cierre de Caja escribe en múltiples tablas Airtable secuencialmente. Si falla en el paso 3 de 5, las tablas quedan en estado inconsistente. Airtable no soporta transacciones ACID — no hay rollback.

Positivo

RBAC 8 roles implementado

lib/auth/permissions.ts con hasPermission() y getDataScope() cubre los 8 roles. Es un diseño sólido — el problema es que solo está aplicado en la capa UI, no en las API routes.

Positivo

Headers de seguridad en next.config

CSP, X-Frame-Options, X-XSS-Protection configurados en next.config.ts. Es una práctica correcta que Créalo implementó proactivamente.

Crítico

LCP 3.5–4.5s en móvil

Sin skeleton loaders, sin lazy loading declarado, sin optimización de imágenes confirmada. Los líderes de tienda usan la PWA en celulares con conexión variable. Google considera LCP crítico >4s — inutilizable en 3G.

Crítico

Sonner cobertura 20%

Solo 20% de los flujos críticos tienen notificaciones Sonner. El 80% restante (incluyendo cierres de caja y rechazos de checklist) falla silenciosamente — sin confirmación de éxito ni feedback de error.

Crítico

Sin persistencia local de checklists

Los formularios de checklist no guardan progreso en localStorage. Si la sesión expira a la mitad de un checklist de 20 preguntas, el usuario pierde todo el progreso sin posibilidad de recuperación.

Alto

Sin estados de error en formularios críticos

Cierre de Caja, Depósitos y Checklists no tienen estados de error. Si Airtable falla, el usuario no sabe si su data fue guardada — riesgo de enviar el cierre de caja dos veces.

Alto

Tailwind v4 OKLch en producción

OKLch es el default en Tailwind v4, pero el soporte no es uniforme en Android 10–11 (prevalentes en Chile). Los colores semánticos pueden renderizar diferente en los dispositivos de los líderes de tienda.

Positivo

PWA instalable + shadcn/ui v4

Los trabajadores pueden instalar la app sin App Store. shadcn/ui v4 proporciona componentes accesibles y consistentes. El diseño visual es profesional para el contexto del cliente.

Hallazgo más urgente de toda la evaluación

Airtable Free tiene 1,000 registros. Con 258 de seed y 182 registros dinámicos/día (36 instancias + 144 respuestas + 2 cierres), los 742 disponibles se agotan en 4.1 días. El cálculo anterior de "3 semanas" no contaba las respuestas individuales por ítem.

Crítico

Observabilidad 10/100

No hay Sentry, no hay logging estructurado, no hay alertas de infra, no hay métricas de negocio, no hay dashboard de estado. El primer aviso de falla en producción llega por WhatsApp.

Crítico

Sin CI/CD formal

Los deploys son manuales vía git push a Vercel. No hay pipeline que corra tests antes del deploy, no hay preview environments, no hay rollback documentado ante un deploy roto.

Crítico

Sin runbooks

No existe documentación de "qué hacer cuando falla X". Si el cron falla a las 7AM: ¿quién lo detecta? ¿cómo se ejecuta manualmente? ¿quién tiene acceso al panel de Vercel? No hay respuestas documentadas.

Alto

Vercel Hobby sin SLA

Vercel Hobby no tiene SLA. Un incidente puede dejar la plataforma caída horas sin compensación ni soporte prioritario. Para una plataforma que reemplaza WhatsApp en operaciones diarias, este es un riesgo real.

Alto

Sin plan de migración DB

Cuando Airtable Team alcance 125,000 filas (~mes 7–10), no hay plan de migración documentado. Migrar a Supabase o PostgreSQL con datos en producción es un proyecto de 2–4 semanas de riesgo alto.

Positivo

Deploy automático con Vercel

La integración git → Vercel con deploy en 30s es una ventaja real frente a AWS EC2. Para un desarrollador solo manteniendo el proyecto, esto elimina la carga operacional de DevOps tradicional.

05 · Defectos S1

8 Defectos Críticos Bloqueantes

Estos defectos deben resolverse antes de que María José dependa de la plataforma para operaciones financieras críticas o supervisión diaria real.

S1-01CRÍTICOPermisos no validados en API routes — IDOR
Los 10 endpoints API validan autenticación pero no autorización. Un Trabajador accede a datos de Finanzas directamente vía API. El roadmap de Créalo lo confirma como "P1 pendiente" — deploy consciente con vulnerabilidad activa.
Fix: Aplicar hasPermission() y getDataScope() (ya implementados en lib/auth/permissions.ts) en cada uno de los 10 API routes antes de consultar Airtable. Tiempo estimado: 1–2 días.
S1-02CRÍTICOCron 7AM falla silenciosamente — timeout 2.25× sobre límite
36 plantillas × 220ms = 7.92s base · 16.2s worst case. Vercel Hobby cancela a los 10s sin retry. El dashboard muestra 0/0 que los supervisores interpretan como "todo correcto" (falso negativo).
Fix: Migrar a Vercel Pro ($20/mes → 60s timeout) o reestructurar el cron en batches de 5 plantillas con queue. La opción A es más rápida.
S1-03CRÍTICOTimezone UTC vs Chile — cron se dispara a hora incorrecta
Vercel ejecuta crons en UTC. Si está configurado a las 7:00 UTC y Chile está en UTC-3/UTC-4, el cron se dispara a las 10–11 AM hora Chile — 3 horas después del inicio del turno de los trabajadores.
Fix: Configurar el cron a las 10:00 UTC (= 7:00 AM UTC-3). Usar America/Santiago como timezone de referencia en toda la lógica temporal.
S1-04CRÍTICOSin protección doble envío en cierre de caja — datos financieros duplicados
En conexiones lentas (habitual en retail), el usuario presiona "Enviar", no ve respuesta inmediata, y presiona de nuevo. Resultado: dos cierres de caja registrados para el mismo día.
Fix: Deshabilitar el botón submit después del primer click. Deduplicar por (fecha, tienda, turno) en la API route antes de escribir a Airtable.
S1-05CRÍTICOFalla Airtable durante cierre de caja — datos del formulario perdidos
Si Airtable retorna error 503 durante el cierre de caja, el formulario muestra error genérico y los datos se pierden. El líder de tienda no tiene copia de lo que ingresó.
Fix: Guardar el estado del formulario en localStorage antes de enviar. Si falla, mostrar modal de recuperación con los datos guardados. Implementar retry automático con backoff exponencial.
S1-06CRÍTICODashboard muestra 0/0 como correcto cuando el cron falla — falso negativo
Cuando el cron falla, el dashboard de David y María José muestra "0 de 0 completados" sin distinción entre "todos completaron" y "ningún checklist fue generado".
Fix: Agregar indicador de estado del cron como métrica separada. Si no corrió a las 7:15 AM ± 30 min, mostrar alerta visual roja en el dashboard de supervisión.
S1-07CRÍTICOSesión expirada durante checklist — todo el progreso perdido
Si la sesión NextAuth expira mientras el trabajador completa un checklist de 20 preguntas, al presionar "Guardar" es redirigido al login. Todo el progreso se pierde sin posibilidad de recuperación.
Fix: Implementar localStorage para guardar el estado del checklist en tiempo real (debounce 2s). Al re-autenticar, ofrecer "Continuar donde dejaste" con los datos guardados.
S1-08CRÍTICOCambio de rol con sesión activa — permisos no actualizados hasta expiración del token
Si un admin cambia el rol de un usuario en Airtable (ej. Trabajador → Jefe_Departamento), el usuario sigue con permisos del rol anterior hasta que el JWT expire.
Fix: Implementar revocación de sesión al cambiar roles. Usar session: { strategy: 'database' } en NextAuth para invalidar sesiones inmediatamente al cambiar permisos.
